# 第四步：换成你自己的业务知识库

`my-notes/` 是业务知识库目录。你可以把里面的示例青橙咖啡资料替换成自己的业务资料。

## 推荐写法

不要把所有内容都塞进一个超长文件。建议按主题拆开：

```text
my-notes/
  index.md
  account.md
  billing.md
  refund.md
  faq.md
```

每个文件只讲一个主题。这样切块后更容易检索到准确资料。

## 每条知识要写清楚

好的知识库内容应该包含明确条件：

```text
订单支付后 30 分钟内可以自助取消。
订单开始配送后不支持取消。
退款会在 1 到 3 个工作日原路返回。
```

不好的写法：

```text
订单一般可以取消，退款很快到账。
```

RAG 需要具体、可引用、可判断边界的资料。

## 修改后要重新导入

每次修改 `my-notes/` 后，都要重新生成向量库：

```bash
npm run ingest
```

否则 `src/query.js` 读到的还是旧的 `vector-store.ndjson`。

## 测试问题

换完知识库后，准备 5 到 10 个测试问题：

```text
如何退款？
发票多久可以开？
会员等级怎么算？
资料里没有答案的问题是什么？
```

每次调整文档、chunk 或 topK 后，都用这些问题回归测试。
